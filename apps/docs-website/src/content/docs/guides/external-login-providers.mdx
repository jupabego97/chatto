Chatto can consume one or more OIDC providers for sign-in. Providers are configured in `chatto.toml` as repeated `[[auth.providers]]` tables, and the login page renders one button per configured provider.

<Aside type="note">
  Chatto acts as an OIDC client here. This does not change Chatto's own OAuth authorization endpoint used by Chatto frontends when they connect to a server.
</Aside>

## Supported providers

| Type   | Use for                                |
| ------ | -------------------------------------- |
| `oidc` | Any standards-compatible OIDC provider |

Direct GitHub, GitLab, Google, and Discord provider adapters are not part of Chatto's external login surface. Use an OIDC provider for external sign-in.

## Provider IDs

Every provider needs a stable `id`. The ID appears in login URLs and audit metadata. OIDC account matching uses the verified issuer URL plus subject, so already-linked OIDC identities do not depend on the provider ID remaining unchanged, but changing it still changes callback URLs and operator-facing metadata.

Use short, URL-safe IDs that describe the local provider slot:

```toml
id = "chatto-hub"
id = "company-idp"
id = "staff-sso"
```

Provider IDs must not contain spaces, `/`, `?`, or `#`.

## Callback URLs

Register this callback URL with each OIDC provider:

```txt
https://chat.example.com/auth/providers/{providerID}/callback
```

For example, a provider with `id = "chatto-hub"` on `https://chat.example.com` uses:

```txt
https://chat.example.com/auth/providers/chatto-hub/callback
```

Chatto derives the callback from `webserver.url`, so set that value to the public HTTPS origin of your server.

For compatibility with older single-provider OIDC deployments, Chatto also registers `/auth/oidc` and `/auth/oidc/callback` when exactly one OIDC provider is configured. New OIDC client registrations should use the provider-specific callback URL.

## Configuration

<Tabs>
<TabItem label="TOML">

```toml
[webserver]
url = "https://chat.example.com"

[[auth.providers]]
id = "chatto-hub"
type = "oidc"
label = "Chatto Hub"
issuer_url = "https://hub.chatto.dev"
client_id = "..."
client_secret = "..."

[[auth.providers]]
id = "company-idp"
type = "oidc"
label = "Company SSO"
issuer_url = "https://id.company.example"
client_id = "..."
client_secret = "..."
```

`issuer_url` must be an absolute `http` or `https` URL. Chatto uses OIDC discovery from that issuer and verifies the ID token before matching the external identity.

</TabItem>
<TabItem label="Environment">

For container platforms, configure providers with counted environment variables. Each provider uses `CHATTO_AUTH_PROVIDERS_<index>_<field>`, with indexes starting at `0` and no gaps:

```bash
CHATTO_AUTH_PROVIDERS_0_ID=chatto-hub
CHATTO_AUTH_PROVIDERS_0_TYPE=oidc
CHATTO_AUTH_PROVIDERS_0_LABEL="Chatto Hub"
CHATTO_AUTH_PROVIDERS_0_ISSUER_URL=https://hub.chatto.dev
CHATTO_AUTH_PROVIDERS_0_CLIENT_ID=...
CHATTO_AUTH_PROVIDERS_0_CLIENT_SECRET=...

CHATTO_AUTH_PROVIDERS_1_ID=company-idp
CHATTO_AUTH_PROVIDERS_1_TYPE=oidc
CHATTO_AUTH_PROVIDERS_1_LABEL="Company SSO"
CHATTO_AUTH_PROVIDERS_1_ISSUER_URL=https://id.company.example
CHATTO_AUTH_PROVIDERS_1_CLIENT_ID=...
CHATTO_AUTH_PROVIDERS_1_CLIENT_SECRET=...
```

Supported fields are `ID`, `TYPE`, `LABEL`, `CLIENT_ID`, `CLIENT_SECRET`, `ISSUER_URL`, `SCOPES`, `REQUEST_EMAIL`, and `PROVIDER_OPTIONS_<KEY>`. `SCOPES` is comma-separated, for example `CHATTO_AUTH_PROVIDERS_0_SCOPES=openid,profile,groups`.

Any `CHATTO_AUTH_PROVIDERS_<index>_*` variables replace providers from `chatto.toml`.

Legacy `CHATTO_AUTH_OIDC_*` variables are not accepted. If any are present, Chatto fails startup with a migration error instead of silently booting without external login providers.

</TabItem>
</Tabs>

## Scopes

Chatto requests these default OIDC scopes:

| Type   | Default scopes               |
| ------ | ---------------------------- |
| `oidc` | `openid`, `profile`, `email` |

You can override scopes per provider:

```toml
[[auth.providers]]
id = "chatto-hub"
type = "oidc"
issuer_url = "https://id.example.com"
client_id = "..."
client_secret = "..."
scopes = ["openid", "profile", "groups"]
```

Chatto always includes `openid` even if you omit it from a custom scope list.

Set `request_email = false` to omit the default email scope. Chatto can create and link OIDC-backed users without receiving an email address.

## Account matching and creation

Chatto matches external logins by OIDC identity, not by email address.

- OIDC identities are matched by verified issuer URL plus subject.
- Email claims are treated as profile data and are not used to pick an account.
- Verified email claims, when present, are attached to the Chatto user after explicit account creation or linking.

If a provider callback returns an identity that is not linked to a Chatto account, Chatto does not silently create an account. It stores a short-lived pending OIDC login and asks the user to choose one of these actions:

- create a new passwordless Chatto account from the OIDC identity
- sign in with an existing local account and link the OIDC identity
- link the OIDC identity to the account they are already signed into
- cancel the login

This avoids accidentally creating a second Chatto account for a user who expected OIDC login to attach to an existing account.

## Discovery fields

Public clients can discover configured providers in two places:

- GraphQL `Server.authProviders` returns provider IDs, types, labels, and login URLs.
- `GET /api/server` returns the same metadata in `authProviders`.

Older clients may still read `/api/server.authMethods`. That legacy field stays method-oriented, so OIDC appears as `oidc` even when the configured provider ID is something like `chatto-hub`.
